## Архитектура провайдеров данных

### Общий интерфейс провайдера:
Каждый провайдер должен реализовывать следующий интерфейс:

```javascript
window.ProviderName = {
    // Инициализация подключения
    requestAccess: async function() { /* возвращает boolean */ },
    
    // Получение списка доступных пар/символов
    getPairs: function() { /* возвращает массив строк */ },
    
    // Загрузка данных
    fetchData: async function(range, pair) { /* возвращает массив свечей */ }
};
```

### Формат свечи:
```javascript
{
    time: number,      // Unix timestamp в секундах
    open: number,      // Цена открытия
    high: number,      // Максимальная цена
    low: number,       // Минимальная цена
    close: number,     // Цена закрытия
    volume: number     // Объем (опционально)
}
```

## Важные детали реализации

### 1. Логирование
Функция `addLog(message)` используется для вывода сообщений в лог-панель. Все провайдеры должны использовать ее для информирования пользователя о статусе операций.

### 2. Обработка ошибок
- Все провайдеры должны иметь fallback на mock данные
- При ошибках API нужно информировать пользователя через `addLog()`
- Возвращать пустой массив `[]` при невозможности получить данные

### 3. Mock данные
При создании mock данных учитывать:
- Реалистичные ценовые уровни для разных типов активов
- Соответствующую волатильность
- Правильные временные интервалы

### 4. Обновление интерфейса
При добавлении нового провайдера необходимо:
1. Добавить файл провайдера
2. Добавить ссылку на скрипт в `index.html`
3. Добавить обработку в `main.js` в функциях `setDataSource()` и `setPair()`
4. Добавить опцию в меню выбора источника

## Рекомендации по внесению изменений

### При добавлении нового провайдера:
1. Проверить, что API доступен и документирован
2. Реализовать стандартный интерфейс провайдера
3. Добавить детальное логирование
4. Реализовать fallback на mock данные
5. Протестировать с разными диапазонами и парами

### При изменении существующего кода:
1. Сохранять обратную совместимость
2. Не нарушать работу других провайдеров
3. Обновлять документацию при изменении интерфейсов
4. Тестировать все источники данных после изменений

### При оптимизации производительности:
1. Кэшировать данные там, где это уместно
2. Минимизировать количество запросов к API
3. Использовать эффективные алгоритмы для расчетов индикаторов

## Типичные проблемы и решения

### 1. Ошибки API:
- Проверить API ключ и лимиты
- Добавить задержки между запросами при лимитах
- Использовать mock данные как fallback

### 2. Проблемы с отображением:
- Проверить формат временных меток (Unix timestamp в секундах)
- Убедиться, что данные отсортированы по времени
- Проверить корректность ценовых данных

### 3. Проблемы с интерфейсом:
- Обновить `window.onresize()` при изменении структуры DOM
- Проверить обработчики событий для новых элементов
- Тестировать на разных размерах экрана
